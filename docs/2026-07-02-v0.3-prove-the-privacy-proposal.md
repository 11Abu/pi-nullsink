# v0.3 proposal — "prove the privacy" (PRE-AUDIT DRAFT)

Status: **AUDITED — REJECTED AS SCOPED** (adversarial audit, 2026-07-02/03). Verdict: Scope D;
F1 reduced to docs + surfacing of pi's EXISTING proxy machinery. Key facts the audit proved:
pi's CLI installs `undici.EnvHttpProxyAgent` as the global dispatcher (`configureHttpDispatcher()`
in dist/cli) and pi has a built-in `httpProxy` setting — under pi's real runtime the extension's own
`fetch` already honors `HTTPS_PROXY`, so ONE user-side setting routes BOTH `/v1` and wallet calls;
the proposed Bun `fetch({proxy})` option is a silent no-op under pi's required Node ≥22 runtime
(undici ignores it) and Bun rejects `socks5://` outright — Scope A as specified would have been the
opposite of fail-closed. F2 is post-hoc-incoherent in a client with no request-interception hook;
F3 contradicts the product ethos. v0.3 direction: publish + harden + document Tor via pi's own
proxy support. Original draft kept below for the record.

v0.2 context: merged to main (6c5f99a), 148 tests green, **npm publish pending the human live
smoke** — zero external users today.

## Problem statements (per candidate feature)

**F1 — Tor/SOCKS routing for wallet calls.**
Problem: nullsink's "no IP logs" is a promise, not a property. Every `/balance`/`/buy`/`/order-status`
call delivers `(user IP, key or key-hash)` to the operator; a compromised or coerced box could
correlate IP↔key and undo the paid-for anonymity. Routing our wallet calls through Tor makes the
correlation impossible at the source rather than contractually declined.
Who has this problem: the exact audience nullsink markets to (threat model includes the operator
and the user's own ISP). Evidence it's wanted: none yet — deduced from the product's own trust-model
doc, not from a user report.

**F2 — Tinfoil attestation verification.**
Problem: the `nullsink-tinfoil` provider's "sealed enclave" property is displayed, not verified; users
trust a label. nullsink documents the attestation flow (docs/tinfoil-attestation.md upstream).
Evidence it's wanted: none; differentiator argument only. Requires a research spike (measurement
chain, what root of trust we pin, what a failed verification even means for an already-sent prompt).

**F3 — Local spend ledger + burn rate.**
Problem: anonymous service ⇒ no server-side history exists anywhere; users cannot answer "what did
I spend yesterday / how long will $18 last". All data already flows through us per-turn.
Tension: writing a permanent local record of usage metadata is philosophically opposite to the
product's ethos; must be off-by-default + incognito-gated, which caps its reach.

## Evidence & monitoring honesty (what we should have observed, and can observe)

- We have **zero usage signal**: unpublished, no external users, live smoke not yet run.
- The product's ethos forbids telemetry; "monitoring" for this project can only ever be:
  GitHub issues/discussions, direct client (nullsink operator) feedback, npm download counts
  (weak), and our own dogfooding pain.
- Signals that WOULD justify each feature: F1 — a user/issue raising IP exposure, or the client

## Scope options

- **A (minimal):** F1 only, wallet calls only. Config `proxy` (+`NULLSINK_PROXY` env), threaded into
  `WalletApi` fetch, **fail-closed** (proxy configured+unreachable ⇒ wallet calls error, never fall
  back to direct). Hub row under Privacy + "test connection" action. README section documents the
  `/v1`-side story (env proxy vars / torsocks) and the boundary list (latency, payment-layer
  separation, timing correlation, exit blocking, future `.onion` upstream ask).
- **B:** A + F2 (attestation) — needs its own research spike first; not plannable today.
- **C:** A + F3 (ledger) — off-by-default, JSONL in the agent dir, incognito-gated, `/nullsink history`.
- **D (null option):** ship nothing new; v0.3 = publish + deferred-minors hardening + refactor pass.

Author's pre-audit lean: A (or D-then-A). B and C deferred until post-publish signal.

## Architecture sketch (scope A)

- `config.ts`: `proxy?: string` in StoredConfigV2 (validate scheme ∈ {socks5, http, https}); precedence
  env `NULLSINK_PROXY` > file, same pattern as key/baseUrl.
- `wallet.ts`: `WalletApi` constructor gains `proxy?: string`; every fetch passes `{ proxy }` (Bun fetch
  option). Fail-closed = no code path constructs a WalletApi without the configured proxy.
- `index.ts`: thread from config; hub Privacy row (edit, validated); "Test proxy" action = `rails()`
  through the proxy with explicit success/failure copy.
- **Verify-on-implementation:** (1) Bun `fetch({proxy})` accepting `socks5://` URLs — if not, document
  Tor's native `HTTPTunnelPort` as the supported endpoint (still zero third-party deps); (2) whether
  Bun/pi honors `HTTPS_PROXY` env for `/v1` traffic (README claim depends on it).
- Explicit non-goals: bundling a Tor client; proxying `/v1` ourselves (Pi owns those sockets);
  SOCKS auth.

## Tradeoffs

| Choice | Cost | Benefit |
| --- | --- | --- |
| Fail-closed proxy | wallet features break when Tor is down | the feature can't silently lie |
| Wallet-only routing (not /v1) | partial coverage; /v1 needs user env setup | zero fragile interception of Pi internals |
| Off-by-default | most users never gain the protection | no surprise breakage; ethos-consistent |
| Ledger off-by-default + incognito-gated (if C) | low adoption of F3 | no metadata betrayal of the core promise |

## SLC self-assessment (to be audited)

- Simple: A is one config field + one fetch option + docs. B is not simple (research spike). C is
  simple mechanically, complicated ethically.
- Lovable: A is lovable to the niche that chose an anonymous proxy — "the wallet never sees my IP"
  is the same sentence as the product's pitch. C's burn-rate line is the lovable bit.
- Complete: A alone is complete for the wallet plane; the /v1 plane is docs-only — audit question:
  does that make A honest-complete or half-complete?
